# فصل ۵: امنیت مدل، Artifact و زنجیره تأمین

<div dir="rtl">

## مدل به‌عنوان دارایی امنیتی

مدل فقط یک فایل خروجی از فرآیند آموزش نیست. مدل یک دارایی حساس است که می‌تواند منطق تصمیم‌گیری، داده آموزشی، دانش سازمانی و ارزش مالکیت فکری را در خود داشته باشد. بنابراین باید مانند یک `Artifact` امنیتی مدیریت شود.

در `MLSecOps` هر مدل باید منشأ، نسخه، هش، امضا، معیارهای ارزیابی، نتایج تست امنیتی و مجوز انتشار مشخص داشته باشد.

## کنترل‌های امنیتی مدل

| کنترل | هدف |
|---|---|
| `Model Artifact Scanning` | شناسایی فایل‌های ناامن مانند `pickle` و کد مخرب |
| `Backdoor Detection` | کشف رفتار شرطی یا `Trigger-based` |
| `Adversarial Robustness Testing` | سنجش مقاومت در برابر ورودی‌های دستکاری‌شده |
| `Model Signing` | تضمین اصالت و جلوگیری از جایگزینی مدل |
| `Provenance Tracking` | ثبت مسیر ساخت مدل از داده تا انتشار |
| `Access Control` | محدودسازی دانلود، بارگذاری و اجرای مدل |

## حداقل الزامات Adversarial Robustness

این بخش برای مدل‌های کلاسیک مانند `Tabular`، `Vision` و `Speech` اهمیت ویژه دارد. در سامانه‌های `LLM`، ارزیابی امنیتی معمولاً با ابزارهایی مانند `Garak` یا `Promptfoo` و تمرکز روی `Prompt Injection`، `Jailbreak` و `Retrieval Leakage` انجام می‌شود.

هدف از `Adversarial Robustness Testing` این است که مشخص شود مهاجم می‌تواند با تغییرات کوچک و کنترل‌شده در ورودی، مدل را به تصمیم اشتباه هدایت کند یا نه. این تست باید بخشی از `Pre-Deployment Security Validation` باشد.

## تعریف مدل تهدید پیش از تست

نتیجه یک تست امنیتی بدون تعریف مدل تهدید ارزش عملیاتی محدودی دارد. پیش از اجرای تست‌های adversarial باید حداقل این پرسش‌ها پاسخ داده شوند:

| پرسش | نمونه | موضوع |
|---|---|---|
| `Who` | مهاجم خارجی، کاربر داخلی، bot یا سرویس خودکار | مهاجم کیست؟ |
| `Where` | APIهای inference، batch scoring، اپلیکیشن یا سرویس متصل | حمله از کجا انجام می‌شود؟ |
| `Goal` | `Evasion`، `Model Extraction` یا اختلال سرویس | هدف مهاجم چیست؟ |
| `Knowledge` | `Black-box` یا `White-box` | مهاجم چه میزان دانش دارد؟ |

## حداقل تست‌های امنیتی

- استفاده از یک `Security Test Suite` نسخه‌دار تا تست‌ها در هر نسخه مدل قابل تکرار باشند.
- اجرای تست‌های adversarial با ابزارهایی مانند `ART`.
- استفاده از روش‌هایی مانند `FGSM` و `PGD` در مدل‌های vision.
- استفاده از روش‌هایی مانند `HopSkipJump` برای شبیه‌سازی سناریوی black-box.
- بررسی `Backdoor` یا triggerهای پنهان، به‌خصوص در مدل‌های pretrained یا transfer learning.
- اجرای تست‌های مقاومت برای مدل‌های چندوجهی (`Multimodal/VLM`) در برابر دستورات بصری خصمانه (`Adversarial Visual-Instructions`) و تزریق OCR با استفاده از فریم‌ورک‌هایی مانند `AVIBench` یا افزونه‌های چندوجهی `ART`.
- مقایسه نتایج با baselineهای امضاشده، نه فقط یک pass/fail ساده.

## خطر فرمت‌های ناامن

برخی فرمت‌های ذخیره مدل مانند `pickle` می‌توانند هنگام بارگذاری باعث اجرای کد شوند. در چنین حالتی، صرفاً باز کردن مدل آلوده ممکن است به اجرای کد دلخواه مهاجم در محیط آموزش یا استقرار منجر شود.

اصل پیشنهادی:

- استفاده از فرمت‌های امن‌تر مانند `safetensors` در صورت امکان
- اسکن همه `Artifact`ها پیش از بارگذاری
- اجرای بارگذاری مدل در محیط ایزوله
- ممنوعیت بارگذاری مدل از منبع ناشناس

> هشدار: خودِ مسیر «تبدیل به فرمت امن» هم می‌تواند سطح حمله باشد. مطالعه «SILENT SABOTAGE» نشان داد یک bot تبدیل `pickle` به `safetensors` در یک پلتفرم عمومی مورد سوءاستفاده قرار گرفت تا artifactهای آلوده تولید کند. بنابراین انتخاب فرمت امن کافی نیست؛ سرویس تبدیل، حساب bot و اعتبارنامه‌های آن نیز باید hardening و audit شوند.

## زنجیره تأمین AI

زنجیره تأمین هوش مصنوعی فقط شامل پکیج‌های نرم‌افزاری نیست. در این زنجیره داده، مدل پایه، `Dataset`، `Notebook`، کد آموزش، وابستگی‌ها، کانتینر، رجیستری مدل و اسکریپت‌های استقرار نیز حضور دارند.

| جزء زنجیره | ریسک |
|---|---|
| مدل پایه | آلودگی، `Backdoor` یا رفتار پنهان |
| دیتاست عمومی | `Data Poisoning` یا برچسب‌گذاری غلط |
| وابستگی نرم‌افزاری | آسیب‌پذیری یا `Typosquatting` |
| `Notebook` | نشت Secret یا اجرای کد ناامن |
| `Model Registry` | جایگزینی مدل سالم با مدل آلوده |
| کانتینر | آسیب‌پذیری سیستم‌عامل یا ابزارهای نصب‌شده |

## آسیب‌پذیری‌های زیرساخت MLOps

پلتفرم‌های `MLOps` — مانند `MLflow`، `ClearML`، `Kubeflow`، registryهای مدل و notebook serverها — خود سطح حمله هستند. تحقیقات امنیتی (از جمله HiddenLayer و Hackstery) نشان داده‌اند:

| پلتفرم / بردار | ریسک | کنترل |
|---|---|---|
| `MLflow` | `LFI`/path traversal، دسترسی به credential cloud | patch، network isolation، auth |
| `ClearML` | compromise agent و poison pipeline | hardening agent، allowlist artifact |
| `Notebook Server` | اجرای کد دلخواه، نشت secret | `NB Defense`، sandbox، egress deny |
| `Model Registry` عمومی | مدل آلوده، pickle RCE | `ModelScan`، allowlist، signing |
| `Prefect/Airflow` در ML | secret در log یا config | secret manager، log sanitization |

حمله «Confused Learning» نشان می‌دهد مهاجم می‌تواند از طریق مدل یا metadata آلوده، کل pipeline آموزش را منحرف کند — حتی پیش از رسیدن مدل به inference.

## امنیت Infrastructure-as-Code برای ML

محیط‌های ML معمولاً روی `Kubernetes`، `Terraform` و cloud managed service اجرا می‌شوند. کنترل‌های `DevSecOps` باید در pipeline ML نیز اعمال شوند:

| ابزار | کاربرد |
|---|---|
| `Checkov` / `tfsec` | اسکن misconfiguration در Terraform |
| `terraform-compliance` | تست negative برای IaC |
| `TFlint` | lint برای Terraform |
| `Kyverno` / `OPA Gatekeeper` | policy در cluster |
| `NetworkPolicy` | محدود کردن egress podهای training/inference |

## SBOM و AI-BOM

`SBOM` اجزای نرم‌افزاری را فهرست می‌کند. در سامانه‌های هوش مصنوعی به اطلاعات بیشتری نیاز داریم که می‌توان آن را `AI-BOM` نامید.

| سند | شامل چه چیزهایی است |
|---|---|
| `SBOM` | پکیج‌ها، نسخه‌ها، وابستگی‌ها و آسیب‌پذیری‌ها |
| `AI-BOM` | داده، مدل پایه، پارامترها، متریک‌ها، تست‌ها، منشأ و شواهد آموزش |

برای تولید `SBOM` می‌توان از ابزارهایی مانند `Syft` و `CycloneDX` استفاده کرد. برای `AI-BOM` نیز باید اطلاعات اختصاصی مدل، dataset، artifactها و اجزای AI ثبت شود؛ ابزارهایی مانند `cdxgen` (دستور `aibom`) و `OWASP AIBOM Generator` در همین حوزه قابل استفاده‌اند.

استاندارد `CycloneDX 1.7` (منتشرشده ۲۰۲۵ و مصوب به‌عنوان نسخه دوم `ECMA-424`) به‌طور رسمی از `ML-BOM` پشتیبانی می‌کند و امکان ثبت مدل به‌عنوان component، hash هر فایل وزن و نسخه dataset را فراهم می‌سازد. توصیه عملی این است که تولید `ML-BOM` بخشی از مرحله promotion مدل باشد — دقیقاً مانند تولید `SBOM` در pipeline ساخت container. دستورها و نمونه‌های اجرایی در فصل ۱۲ آمده است.

## معیارهای پذیرش امنیتی

پیش از انتشار مدل، باید معیارهای عددی و قابل بررسی تعریف شود.

> توجه مهم: اعداد جدول زیر صرفاً **نمونه** هستند و باید بر اساس مدل تهدید، حساسیت کاربرد و baseline واقعی هر سازمان تنظیم شوند. این مقادیر یک نسخه واحد برای همه سامانه‌ها نیستند؛ برای یک مدل credit scoring پرریسک ممکن است سخت‌گیرانه‌تر و برای یک ابزار داخلی کم‌ریسک ممکن است نرم‌تر باشند.

| معیار | شرط پذیرش نمونه |
|---|---|
| `Attack Success Rate` | نسبت به baseline بیش از ۲٪ افزایش نداشته باشد |
| `Clean Accuracy Drop` | افت دقت روی داده سالم بیش از ۱٪ نباشد |
| `Transfer Rate` | نرخ موفقیت حمله ساخته‌شده روی surrogate در برابر مدل اصلی حداکثر ۵٪ باشد |
| `Backdoor Activation` | فعال‌سازی رفتار پنهان باید صفر باشد |
| `Critical Vulnerability` | هیچ آسیب‌پذیری بحرانی در `Artifact` مجاز نیست |
| امضای مدل | مدل و شواهد باید با کلید معتبر امضا شده باشند |

اگر هر یک از معیارها از آستانه تعیین‌شده عبور کند، مدل نباید وارد `Production` شود و باید hardening یا retraining انجام شود.

## Provenance و امضا

هر مدل منتشرشده باید قابل ردیابی باشد. باید بتوان پاسخ داد:

- مدل از کدام داده ساخته شده است؟
- چه نسخه‌ای از کد آموزش استفاده شده است؟
- چه کسی انتشار را تأیید کرده است؟
- کدام تست‌ها اجرا شده‌اند؟
- هش و امضای مدل چیست؟
- آیا مدل در مسیر انتشار تغییر کرده است؟

برای امضای مدل، علاوه بر ابزارهای عمومی مانند `Cosign`، اکنون ابزار اختصاصی `sigstore/model-transparency` (پکیج `model-signing`) وجود دارد که با همکاری `OpenSSF`، `NVIDIA` و `HiddenLayer` برای مدل‌های ML ساخته شده است. این ابزار امضای keyless مبتنی بر هویت (`OIDC`) را پشتیبانی می‌کند، نیاز به مدیریت کلید بلندمدت را حذف می‌کند و رویداد امضا را در transparency log عمومی `Rekor` ثبت می‌نماید تا قابل ممیزی باشد (دستورها در فصل ۱۲).

در برخی سناریوها می‌توان از `Watermarking` نیز برای درج نشان قابل شناسایی در مدل یا خروجی آن استفاده کرد. این مکانیزم می‌تواند برای اثبات مالکیت، شناسایی مدل‌های کپی‌شده یا ردیابی استفاده غیرمجاز مفید باشد، اما کنترل کمکی است و جایگزین امضای دیجیتال یا `Provenance Tracking` نمی‌شود.

## خروجی ارزیابی امنیتی

در پایان ارزیابی امنیتی، باید یک `Security Evidence Pack` همراه مدل ذخیره شود. این بسته حداقل شامل موارد زیر است:

- گزارش کامل تست‌های adversarial
- نتایج متریک‌هایی مانند `ASR`، `Accuracy` و `Clean Accuracy Drop`
- نسخه دقیق `Security Test Suite`
- هش و امضای دیجیتال مدل
- مستندات مدل تهدید تعریف‌شده
- attestation مربوط به فرآیند ساخت و انتشار

## Federated Learning

اگر سامانه از `Federated Learning` استفاده می‌کند، تهدیدهای خاصی نیز مطرح می‌شود. در این معماری داده خام روی نودها باقی می‌ماند و فقط updateهای مدل مبادله می‌شود؛ اما یک نود مخرب می‌تواند update آلوده ارسال کند یا تلاش کند اطلاعات سایر شرکت‌کنندگان را استنتاج کند.

| کنترل | هدف |
|---|---|
| `Secure Aggregation` | سرور مرکزی فقط نتیجه تجمیع‌شده updateها را ببیند. |
| `Client Attestation` | هویت و وضعیت امنیتی نودهای مشارکت‌کننده بررسی شود. |
| مقاومت در برابر `Byzantine Attacks` | updateهای مخرب یا نادرست اثر محدود داشته باشند. |
| `Audit Trail` | مشارکت نودها، updateها و تصمیمات تجمیع ثبت شوند. |

## مدیریت کلیدها و Secrets

مدل‌ها و پایپ‌لاین‌های AI معمولاً با کلیدهای API، credentialهای `Model Registry`، کلیدهای امضای مدل و tokenهای سرویس‌های ابری کار می‌کنند. این اطلاعات نباید در کد، notebook، image، فایل تنظیمات یا bundle مدل ذخیره شوند.

کنترل‌های پیشنهادی:

- استفاده از `Secret Manager` یا `KMS` مانند `Vault`، `AWS Secrets Manager`، `AWS KMS`، `Azure Key Vault` یا `Google Cloud KMS`
- proxy gateway برای کلید API مدل (الگوی `BlackVault`): agent و pipeline هرگز کلید خام نمی‌بینند؛ kill switch فوری در صورت compromise
- اعمال اصل کمترین دسترسی برای همه credentialها
- نگهداری کلیدهای امضای مدل خارج از repository و pipeline
- ثبت تمام عملیات مشاهده، استفاده، چرخش و حذف secretها
- چرخش و ابطال سریع کلیدها در صورت افشا یا خروج سرویس از چرخه عملیاتی

## اصل عملی

هیچ مدل ناشناس، بدون امضا، بدون منشأ و بدون تست امنیتی نباید وارد `Production` شود. مدل باید همان‌قدر قابل کنترل باشد که یک تصویر کانتینر یا پکیج نرم‌افزاری حساس قابل کنترل است.
